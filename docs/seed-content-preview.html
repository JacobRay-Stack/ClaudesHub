Auto-Generate Git Commit Messages

Staff Pick Skill Git & Review
git automation claude-code developer
47 upvotes 312 downloads 8 comments

Description

A skill that analyzes your staged changes and writes a conventional commit message.

What It Does

When you run /commit, this skill reads your git diff, understands the intent of the changes, and generates a commit message following Conventional Commits format (feat:, fix:, refactor:, etc.). It includes a short subject line and a body explaining the "why" when the change is non-trivial.

Installation

Copy the skill folder to your skills directory:

~/.claude/skills/commit-generator/

Usage

Stage your changes, then tell Claude:

  • "commit this"
  • "write a commit message for my changes"
  • /commit

The Code

# commit-generator SKILL.md
---
name: commit-generator
description: Generate conventional commit messages from staged changes
triggers:
  - /commit
  - commit this
---

Analyze the current `git diff --staged` output. Write a commit message following
Conventional Commits:

1. Determine the type: feat, fix, refactor, docs, test, chore, style, perf
2. Write a subject line under 72 characters (imperative mood)
3. If the change is non-trivial, add a body explaining WHY, not what
4. If multiple concerns, suggest splitting into separate commits

Format:
```
type(scope): subject

Body explaining the motivation and context.
```

Do NOT include file lists. Focus on intent and impact.

Compatibility

Claude Code v1.x+. Works on Mac, Windows, Linux. Requires git installed.

CLAUDE.md for Next.js + TypeScript + Supabase

Staff Pick CLAUDE.md Code & Refactoring
nextjs typescript tailwind supabase developer
63 upvotes 489 downloads 12 comments

Description

A battle-tested CLAUDE.md for Next.js App Router projects using TypeScript, Tailwind CSS v4, and Supabase.

Stack

Next.js 16, React 19, TypeScript, Tailwind CSS v4, Supabase (auth, database, storage)

What It Configures

  • Enforces App Router conventions (page.tsx, layout.tsx, server components by default)
  • Requires TypeScript strict mode patterns
  • Uses Tailwind utility classes, no CSS modules
  • Supabase client created via server/client helpers, never raw imports
  • RLS policies required for every new table
  • Server actions for mutations, not API routes

The File

# Project CLAUDE.md

## Tech Stack
- Next.js 16 (App Router), React 19, TypeScript
- Tailwind CSS v4 with CSS variables for theming
- Supabase (auth, Postgres, storage)

## Conventions
- All components in src/components/, grouped by feature
- All pages in src/app/ following App Router conventions
- Server components by default. Add "use client" only when needed
- Use createServerClient() from @/lib/supabase/server for server-side
- Use createBrowserClient() from @/lib/supabase/client for client-side
- Never import supabase-js directly

## Code Style
- TypeScript strict. No `any`.
- Tailwind for all styling. No CSS modules.
- Use `cn()` utility for conditional classes
- Prefer server actions over API routes for mutations

## Database
- Every new table needs RLS policies
- Use migrations in supabase/migrations/
- Foreign keys reference profiles(id) for user relations

## Don't
- Don't use pages router patterns
- Don't skip RLS policies
- Don't use inline styles
- Don't commit .env files

Customization

Swap the Supabase references for Prisma/Drizzle if you're not using Supabase. The Next.js and Tailwind conventions apply regardless of your database choice.

Supabase MCP Server — Full Database Access

Staff Pick MCP Server Data & Analytics
mcp supabase sql database
89 upvotes 1,204 downloads 23 comments

Description

Connect Claude Code directly to your Supabase project. Query tables, inspect schema, run migrations, and manage storage without leaving the terminal.

Config — copy to settings.json
{
  "mcpServers": {
    "supabase": {
      "command": "npx",
      "args": [
        "-y",
        "@supabase/mcp-server",
        "--project-ref",
        "YOUR_PROJECT_REF"
      ]
    }
  }
}

Environment Variables

  • SUPABASE_ACCESS_TOKEN — Get from supabase.com/dashboard/account/tokens

Available Tools

  • list_tables — List all tables in your database
  • execute_sql — Run any SQL query
  • get_table_schema — Inspect columns, types, and constraints
  • list_extensions — See installed Postgres extensions
  • apply_migration — Run a migration file
  • list_storage_buckets — See storage buckets

Token Usage

  • Init: ~8,000 tokens (schema loaded on startup)
  • Per call: ~500–2,000 tokens depending on result size
  • Tip: If you have 50+ tables, init tokens increase significantly. Use LIMIT in queries.

Setup Steps

  • 1. Go to supabase.com/dashboard/account/tokens
  • 2. Generate a new access token
  • 3. Set it: export SUPABASE_ACCESS_TOKEN=your_token
  • 4. Add the config above to your settings.json
  • 5. Replace YOUR_PROJECT_REF with your project ref (found in project settings)
  • 6. Restart Claude Code

Example Prompts

  • "Show me all tables in the database"
  • "Query the resources table for the top 10 by upvote count"
  • "Add a reputation column to the profiles table"
  • "Create a migration that adds a notifications table"

Gotchas

  • The project ref is NOT your API URL. It's the short string like "rynqjcskoburetbftplk"
  • execute_sql runs with full privileges. Be careful with DELETE and DROP
  • Large result sets can eat tokens fast. Use LIMIT in your queries
  • On Windows, you may need to use the full path to npx

Auto-Format Files on Save

Hook Code & Refactoring
automation prettier eslint formatting
31 upvotes 156 downloads 5 comments

Description

Automatically runs Prettier on any file Claude Code writes or edits. No more inconsistent formatting.

Trigger

PostToolUse — fires after the Write or Edit tool completes.

Configuration

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hook": "npx prettier --write \"$CLAUDE_FILE_PATH\""
      }
    ]
  }
}

How It Works

  • 1. Claude Code writes or edits a file
  • 2. The hook fires and runs Prettier on that specific file
  • 3. The formatted result is what gets saved
  • 4. Claude sees the formatted version on its next read

This means you never have to say "format this" — it happens automatically every time Claude touches a file.

Requirements

  • Prettier installed in your project: npm install -D prettier
  • A .prettierrc config file (or Prettier defaults are fine)
  • Node.js 18+

Multi-Agent Workflows with Claude Code

Guide Code & Refactoring
claude-code multi-agent workflow tutorial
55 upvotes 14 comments

Description

How to run multiple Claude Code instances in parallel to ship features faster.

Who This Is For

Developers comfortable with Claude Code who want to parallelize their work. You should already know what skills, hooks, and CLAUDE.md files are.

Why Multi-Agent?

A single Claude Code session is fast. But some tasks have independent parts — frontend and backend, tests and implementation, docs and code. Running them in parallel cuts your total time significantly.

The Pattern

  • 1. Create a git branch for each task
  • 2. Open separate terminals (or tmux panes)
  • 3. Run Claude Code in each terminal on its own branch
  • 4. Merge when both are done

Step 1: Set Up Branches

git checkout -b feature/api-endpoints
git checkout main
git checkout -b feature/frontend-ui

Step 2: Run Claude Code in Parallel

# Terminal 1:
cd my-project && git checkout feature/api-endpoints
claude "Build the REST API endpoints for user profiles"

# Terminal 2:
cd my-project && git checkout feature/frontend-ui
claude "Build the profile page UI with these components..."

Step 3: Merge

git checkout main
git merge feature/api-endpoints
git merge feature/frontend-ui

Tips & Gotchas

  • Give each instance a clear, scoped task. Don't let them overlap on the same files.
  • Use separate branches to avoid conflicts.
  • Each instance has its own context window — they don't share memory.
  • If you're on Claude Max, watch your rate limits. Two instances burn tokens 2x faster.
  • The git worktree approach is even cleaner — avoids checkout switching entirely.

SQL Query Builder from Natural Language

Prompt Data & Analytics
sql database analytics prompt-engineering
28 upvotes 6 comments

Description

A prompt pattern that turns plain English questions into optimized SQL queries for your specific database schema.

When to Use

When you need to query a database but don't want to write SQL by hand. Works best when Claude has access to your schema via MCP or CLAUDE.md.

The Prompt

I need a SQL query for my Postgres database.

Schema context: [paste your table definitions or say "use the connected database"]

Question: [your natural language question]

Requirements:
- Use explicit column names, no SELECT *
- Add comments explaining non-obvious joins
- Use CTEs for complex queries instead of nested subqueries
- Include LIMIT if the result set could be large
- Format the SQL for readability

Example Output

Input: "Which users have posted the most resources in the last 30 days?"

-- Top contributors by resource count in last 30 days
WITH recent_resources AS (
  SELECT
    author_id,
    COUNT(*) as resource_count
  FROM resources
  WHERE created_at >= NOW() - INTERVAL '30 days'
  GROUP BY author_id
)
SELECT
  p.username,
  p.display_name,
  rr.resource_count
FROM recent_resources rr
JOIN profiles p ON p.id = rr.author_id
ORDER BY rr.resource_count DESC
LIMIT 20;

Variations

  • Add "explain the query" to get a plain English breakdown
  • Add "optimize for performance" to get index suggestions
  • Add "make it a migration" to get a CREATE VIEW or function instead
